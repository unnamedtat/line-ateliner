# Line Atelier 项目简历与面试沉淀

## 1. 项目一句话定位
Line Atelier 是一个浏览器端图片转线稿与动态笔触生成工具，支持多算法线稿生成、纸张纹理合成、实时参数调节，以及 MP4/GIF/PNG 导出。

## 2. 大厂前端简历怎么写

### 2.1 项目标题写法
Line Atelier（浏览器端线稿动效生成与导出平台）

### 2.2 技术栈写法
React + TypeScript + Vite + p5.js + Web Worker + Canvas 2D + GIF 编码链路

### 2.3 职责写法（可直接粘贴）
1. 负责搭建 React 壳层与 legacy 绘制引擎桥接方案，统一 UI 状态同步与控制事件分发，支持输入分析、纸张纹理、笔触与导出模块联动。
2. 设计并实现图片预处理与渲染计算 Worker 化方案，将主线程高开销任务迁移至 Worker，降低交互卡顿并提升长任务稳定性。
3. 构建多格式导出链路，完成 MP4/GIF/PNG 的统一导出流程、状态管理与异常恢复提示，支持导出进度与失败回退。
4. 负责导出阶段的渲染节奏控制与临时分辨率切换，确保导出结果与预览一致，并在导出后可靠恢复现场渲染状态。
5. 落地工程化保障：TypeScript 类型约束、ESLint/Prettier 规范、Vitest 单测与 Playwright 冒烟测试，形成可持续迭代基础。

### 2.4 成果写法（建议按真实数据替换）
1. 将核心计算任务 Worker 化后，复杂场景下交互阻塞显著下降，导出过程可用性提升。
2. 建立导出失败恢复机制后，用户可自主切换降级路径（如改导出格式/参数）完成交付。
3. 通过统一桥接层改造，降低 legacy 与 React 双栈维护成本，减少跨层联动回归问题。

### 2.5 简历高分版本（4 条精炼）
1. 主导浏览器端线稿动效平台建设，串联图像分析、路径生成、纹理合成与导出闭环。
2. 设计 Worker 异步计算架构（图片预处理/渲染构建），缓解主线程瓶颈并提升动画调参流畅度。
3. 实现 MP4/GIF/PNG 多格式导出系统，支持进度可视化、错误恢复与参数化质量控制。
4. 推动 React 壳层与 legacy 引擎解耦，形成可测试、可维护、可国际化的前端工程体系。

## 3. 项目难点与卡点（面试重点）

### 3.1 难点一：双栈架构协同（React 壳层 + legacy 引擎）
问题：UI 在 React，核心渲染逻辑在 legacy 脚本，直接互调容易造成状态漂移和事件时序错乱。

解决：
1. 通过统一 bridge 抽象快照与动作，保证 UI 只消费快照、只触发动作。
2. 用事件驱动同步 UI 状态，避免组件直接依赖底层全局变量。
3. 将默认值与可见性规则集中，降低跨模块条件分支复杂度。

可讲收益：降低跨层耦合，减少“看起来改了 UI、实际上没改渲染状态”的线上问题。

### 3.2 难点二：浏览器端性能与流畅度
问题：图像分析、边缘构建、变体重建属于 CPU 密集型，主线程执行会影响拖动滑杆和交互响应。

解决：
1. 引入 image/render Worker，将高开销任务迁移至异步线程。
2. 设计请求序列号和 pending map，保障并发请求下的响应对应关系。
3. 明确失败回退路径，Worker 不可用时自动退回主线程，保证功能可用性。

可讲收益：性能优化不是只追求最快，而是“快且可回退”。

### 3.3 难点三：导出链路稳定性
问题：导出涉及帧循环、编码、大内存对象、浏览器能力差异，最容易出现失败和不可复现问题。

解决：
1. 统一导出状态机，导出前做能力判断，导出中持续进度反馈，失败后给恢复动作。
2. 导出时冻结关键参数与资源快照，避免用户调参导致导出结果漂移。
3. 对画布尺寸做临时切换与恢复，确保导出分辨率和编辑态互不污染。
4. 对 GIF 采用 workerScript 方案减少主线程编码压力。

可讲收益：提升“可交付性”，不是只看 demo 成功一次，而是高概率稳定导出。

### 3.4 难点四：兼容性与降级策略
问题：不同浏览器对编码、Worker、后台页面行为支持差异大，手机端尤其明显。

解决：
1. 能力检测前置，提前提示不支持场景，减少无效等待。
2. 构建多路径导出与失败恢复文案，给用户明确下一步操作。
3. 在导出 UI 增加强提示，约束关键使用条件（如保持页面前台）。

可讲收益：把“技术限制”转成“可理解的产品行为”，减少用户困惑和投诉。

## 4. 常见卡点复盘（可作为自我评价）
1. 卡点：并发任务返回顺序不稳定导致结果错位。
处理：请求 ID + pending map + 统一 resolve/reject 回收。

2. 卡点：导出过程中参数变化导致帧内容不一致。
处理：导出前冻结快照，导出期间只读快照。

3. 卡点：高分辨率导出后编辑画布尺寸被污染。
处理：导出期间临时 resize，结束后恢复尺寸与展示样式。

4. 卡点：某些端能力缺失造成“按钮可点但必失败”。
处理：能力探测前置 + 状态文案可解释 + 恢复动作直达。

5. 卡点：legacy 全局状态与 React 状态不一致。
处理：建立单向桥接协议，React 仅读快照，不直接写底层状态。

## 5. 面试问答题库（附参考回答）

### 5.1 项目理解类
Q1：这个项目的核心业务价值是什么？
A：把“图片转线稿 + 动态笔触 + 导出分享”放在同一个浏览器工作台，降低创作门槛，提升从调参到交付的效率。

Q2：为什么要保留 React + legacy 双栈，而不是一次性重写？
A：核心算法与渲染资产已在 legacy 中稳定沉淀，一次性重写风险大。先通过桥接层解耦 UI 与引擎，确保功能持续迭代，再按模块渐进替换。

Q3：你在这个项目中的最关键贡献是什么？
A：统一导出链路与 Worker 化性能方案，让“可用性与可交付性”显著提升。

### 5.2 架构设计类
Q4：桥接层如何避免状态不同步？
A：采用快照读取 + 动作分发模式，并通过统一事件触发 UI 刷新，避免 React 直接读写分散的全局变量。

Q5：为什么 Worker 请求要有序列号？
A：异步并发下响应可能乱序，序列号是将响应与请求绑定的最小可靠机制。

Q6：什么时候不用 Worker？
A：能力不满足、任务粒度太小或传输成本高于计算收益时，直接主线程执行并保留回退。

Q7：如何保证导出与预览一致？
A：导出使用冻结快照，统一帧推进逻辑与导出时渲染配置，避免实时调参污染导出结果。

### 5.3 性能优化类
Q8：你做性能优化的原则是什么？
A：优先优化用户感知延迟（交互流畅度），其次是吞吐；同时保证回退路径，避免“优化后更脆弱”。

Q9：大图处理的主要瓶颈在哪里？
A：像素级扫描、边缘构建和多变体重建是 CPU 热点，主线程执行会造成明显掉帧。

Q10：如何评估优化是否有效？
A：看三类指标：交互延迟、导出成功率、错误恢复完成率，并结合典型机型做对比验证。

### 5.4 导出与兼容类
Q11：为什么 MP4/GIF 两条链路都要做？
A：MP4 更适合清晰交付，GIF 适合社媒传播。双链路能覆盖更多使用场景与平台限制。

Q12：导出最容易踩的坑是什么？
A：浏览器能力差异与页面生命周期（切后台、锁屏、节流）导致的不确定失败。

Q13：你如何处理导出失败体验？
A：不是只弹错误，而是提供可执行恢复动作，例如降参数、改格式、重试入口。

Q14：为什么要在导出界面加前台提醒？
A：这是真实约束条件，提前告知能显著减少“误操作导致失败”的无效重试。

### 5.5 工程化与协作类
Q15：如何保证这类前端图形项目的可维护性？
A：模块边界清晰、类型约束、测试覆盖关键路径、错误可观测、降级可执行。

Q16：你会优先补哪些测试？
A：导出状态机、能力探测分支、Worker 回退分支、桥接层事件同步。

Q17：如何做版本迭代而不破坏创作者体验？
A：保持默认参数稳定、提供可解释变更说明、回归测试覆盖主要风格模板。

### 5.6 深挖追问类
Q18：如果让你继续优化，第一步做什么？
A：补导出链路端到端指标埋点，先定位高频失败路径，再做针对性优化。

Q19：如果浏览器不支持关键能力，你怎么向产品解释？
A：用“能力矩阵 + 降级策略 + 用户提示”三件套，明确边界、成本和替代方案。

Q20：这个项目最体现你能力的一点是什么？
A：在复杂前端图形场景里，把性能、稳定性、工程化和用户体验同时落地。
